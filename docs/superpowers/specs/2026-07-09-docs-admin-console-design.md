# 文档控制台设计稿

## 背景

当前项目已经有两套实际工作流：

1. `site/docs` 直改流
   - 直接修改 `site/docs/**/*.md`
   - 执行 `npm run docs:sidebar` 或 `npm run docs:refresh`
   - 生成 `site/docs/.vitepress/dist`
   - 部署到 `root@121.199.175.111:/www/wwwroot/help.beta.ztocc.com`

2. `pipeline` 全流程
   - 从钉钉拉取文档
   - 清洗/优化/生成 VitePress 文档
   - 构建站点
   - 视需要部署到线上

当前这些流程都依赖手动执行命令，使用成本较高，也容易出现“漏掉一步”“跑错目录”“部署错机器”的问题。目标是提供一个本机网页控制台，把常用流程收拢成稳定可点击的操作入口，同时保留终端可调用能力。

## 目标

- 在现有 `site/server.js` 基础上新增一个本机管理控制台页面
- 控制台只在本机使用，不对外开放，不额外做登录
- 所有生成、拉取、优化、构建任务都在本机执行
- 只有部署步骤通过 SSH 上传到服务器
- 一次只允许执行一个任务，避免目录和产物互相覆盖
- 同一套任务能力既能被网页按钮调用，也能被终端命令调用

## 非目标

- 不做任意 shell 命令执行器
- 不做多任务并行或任务队列
- 不做远程服务器侧的任务执行
- 不在第一版里做复杂权限体系
- 不把现有文档编辑器改造成完整 CMS

## 方案对比

### 方案 A：挂在现有 `site/server.js` 下的 `/admin`

- 复用现有 Express 服务
- 在现有站点里增加管理页面和任务 API
- 任务由 Node 子进程统一调度
- 使用固定按钮触发固定命令

优点：

- 改动最小，最贴近现有项目结构
- 已有 `server.js` 和 `spawn` 使用基础，可直接扩展
- 后续增加按钮和任务成本低

缺点：

- 管理能力和文档服务耦合在同一个进程里
- 需要小心区分普通文档访问和管理接口

### 方案 B：单独管理服务

- 新开一个本地管理服务，例如 `site/admin-server.js`
- 文档站和管理台分离

优点：

- 更隔离
- 对现有文档站影响更小

缺点：

- 多一个服务、多一个端口、多一套启动方式
- 用户认知和运维成本更高

### 方案 C：终端菜单/TUI

- 通过终端交互菜单执行任务
- 不提供浏览器页面

优点：

- 实现快
- 更贴近命令行工作流

缺点：

- 不符合“网页点一下”的主要诉求
- 日志、状态、任务入口展示不如网页直观

## 结论

采用 **方案 A**。

理由：

- 与现有代码最兼容
- 能最快落地网页控制台
- 同时保留通过后端封装能力给终端复用的空间

## 总体架构

在现有 `site/server.js` 中引入三层能力：

1. 管理页面层
   - 提供 `/admin`
   - 用一个简单静态页面展示任务分组、按钮、状态和日志

2. 任务调度层
   - 维护单例任务状态
   - 封装固定任务定义与命令映射
   - 负责启动、停止、状态变更、日志缓存

3. 任务执行层
   - 使用 `spawn()` 执行本机命令
   - 部署任务内部使用 `tar + scp + ssh`
   - 统一收集 stdout/stderr

## 任务模型

系统一次只允许存在一个“运行中任务”。

任务对象统一结构：

```json
{
  "id": "task-20260709-001",
  "key": "docs-refresh",
  "label": "刷新侧边栏并构建站点",
  "status": "idle|running|success|failed|stopped",
  "startedAt": "2026-07-09T10:00:00+08:00",
  "finishedAt": "2026-07-09T10:01:23+08:00",
  "exitCode": 0,
  "logs": ["..."],
  "meta": {}
}
```

约束：

- 有任务运行时，任何新任务启动请求直接返回冲突错误
- 页面按钮在前端禁用，后端仍然做强校验
- 停止按钮只对当前运行任务生效

## 第一版任务清单

### `site/docs` 流程

- `docs-sidebar`
  - 命令：`npm run docs:sidebar`
  - 目录：`site/`
  - 作用：重生成 `sidebar-data.mjs`

- `docs-build`
  - 命令：`npm run build`
  - 目录：`site/`
  - 作用：构建 VitePress

- `docs-refresh`
  - 命令：`npm run docs:refresh`
  - 目录：`site/`
  - 作用：重生成侧边栏并构建

### `pipeline` 流程

- `pipeline-crawl`
  - 命令：`python3 src/dws_crawler.py`
  - 目录：项目根目录
  - 作用：从钉钉拉取文档和图片

- `pipeline-build`
  - 命令：`python3 src/pipeline.py --source ./output`
  - 目录：项目根目录
  - 作用：执行过滤、优化、站点生成

- `pipeline-deploy-fast`
  - 命令：`python3 src/pipeline.py --source ./output --deploy fast`
  - 目录：项目根目录
  - 作用：按现有 pipeline 能力执行快速部署

### 部署与校验

- `deploy-site-dist`
  - 命令：封装当前确认可用的 `tar + scp + ssh` 组合命令
  - 目录：项目根目录
  - 目标：`root@121.199.175.111:/www/wwwroot/help.beta.ztocc.com`
  - 作用：将 `site/docs/.vitepress/dist` 覆盖部署到正式环境

- `verify-online`
  - 命令：`curl` 检查首页或指定关键页面
  - 作用：确认线上内容是否更新

### 一键流程

- `full-site-flow`
  - 执行顺序：
    1. `docs-sidebar`
    2. `docs-build`
    3. `deploy-site-dist`
    4. `verify-online`

- `full-pipeline-flow`
  - 执行顺序：
    1. `pipeline-crawl`
    2. `pipeline-build`
    3. `deploy-site-dist` 或 `pipeline-deploy-fast`
    4. `verify-online`

说明：

- 第一版中“一键流程”仍然是单任务内部的串行子步骤，而不是多个可并行任务
- 一键流程需要把每个子步骤日志完整地串接到同一任务输出中

## 页面设计

页面挂载到 `/admin`，由后端直接返回一个静态 HTML 页面即可，不引入额外前端框架。

页面分区：

1. 当前状态
   - 当前任务名称
   - 运行状态
   - 开始时间
   - 最近一次完成状态

2. `site/docs` 流程
   - 生成侧边栏
   - 构建站点
   - 一键刷新站点

3. `pipeline` 流程
   - 从钉钉拉取
   - 执行 pipeline
   - 一键全流程

4. 部署与校验
   - 部署到正式服务器
   - 查看线上版本
   - 校验指定页面

5. 日志区
   - 实时滚动日志
   - 支持手动清空页面显示
   - 保留最近一次任务完整日志

6. 操作区
   - 停止当前任务
   - 刷新状态

## API 设计

### `GET /api/admin/status`

返回当前任务状态和最近一次任务摘要。

### `GET /api/admin/logs`

返回当前任务或最近一次任务日志。

### `POST /api/admin/tasks/:taskKey`

启动固定任务。

请求示例：

```json
{
  "args": {}
}
```

返回：

- 成功：任务已启动
- 失败：已有任务运行中 / 任务不存在 / 环境校验失败

### `POST /api/admin/stop`

停止当前运行任务。

### `GET /api/admin/tasks`

返回所有可用任务定义，供页面初始化按钮分组和文案。

## 命令封装策略

不在 API 中接受任意命令字符串，而是维护一份固定任务表，例如：

```js
const TASKS = {
  'docs-sidebar': {
    label: '生成侧边栏',
    cwd: SITE_DIR,
    command: 'npm',
    args: ['run', 'docs:sidebar']
  }
}
```

优点：

- 避免命令注入
- 页面和后端只围绕固定任务 key 通信
- 后续扩展行为更可控

## 终端复用方案

网页按钮和终端入口复用同一套任务定义，但第一版不强依赖“调用 HTTP API”的 CLI。

建议分两步：

1. 第一版先把任务定义抽到单独模块，例如 `site/admin-tasks.js`
2. 第二版如有需要，再补一个终端入口，例如：
   - `node admin-cli.js docs-refresh`
   - 或 `npm run admin:task -- docs-refresh`

这样终端和网页使用同一套任务映射，不会出现“网页按钮和命令实际不一致”的问题。

## 安全与约束

### 访问约束

- 服务只监听 `127.0.0.1`
- `/admin` 与管理 API 仅本机可访问
- 不再依赖当前的 JWT 登录机制

### 执行约束

- 禁止任意命令执行
- 禁止并行任务
- 部署目标主机和目录写死在任务配置中
- 运行目录固定，不允许通过请求参数覆盖

### 文件约束

- 部署前默认保留目标目录中的 `.well-known`
- 控制台不负责直接在线编辑任意文件
- 第一版不在控制台里暴露本地文件浏览和写入能力

## 错误处理

需要明确区分以下错误类型：

1. 参数错误
   - 未知任务 key
   - 不支持的校验参数

2. 状态冲突
   - 当前已有任务运行

3. 环境错误
   - `npm`、`python3`、`ssh`、`scp` 不可用
   - `site/docs/.vitepress/dist` 不存在

4. 执行失败
   - 子进程退出码非 0
   - 远程部署命令失败
   - `curl` 校验未通过

返回格式应包含：

- `errorCode`
- `message`
- `taskId`（如适用）

## 日志与状态

日志保留策略：

- 当前运行任务的完整日志常驻内存
- 最近一次完成任务的完整日志保留
- 前端轮询获取，不强依赖 WebSocket

状态更新策略：

- 前端每 1 到 2 秒轮询一次 `/api/admin/status`
- 日志区同样轮询 `/api/admin/logs`

原因：

- 第一版实现简单
- 不引入额外连接状态管理
- 对本机单用户控制台已经足够

## 测试策略

### 自动化测试

第一版至少补这些测试：

1. 任务表测试
   - 关键任务存在
   - 工作目录和命令映射正确

2. 单任务锁测试
   - 运行中再次触发返回冲突

3. 停止任务测试
   - 可向当前子进程发送停止信号

4. API 基本测试
   - `status`
   - `tasks`
   - 启动固定任务

### 手动验证

完成前至少验证：

1. `/admin` 页面可以打开
2. `生成侧边栏` 可执行并看到日志
3. `构建站点` 可执行并成功结束
4. `部署到正式服务器` 能部署到 `121.199.175.111`
5. `查看线上版本` 或校验按钮能拿到线上结果
6. 运行任务时其他按钮被禁用
7. `停止当前任务` 对长任务有效

## 风险与后续演进

### 当前风险

- `dws_crawler.py` 是否需要额外上下文或登录环境，需要在本机环境中再确认
- 一键全流程中的“部署步骤”如果失败，前序产物仍可能已经更新，需要清楚展示失败节点
- 现有 `server.js` 仍保留认证与编辑接口，后续需决定是保留、下线还是与 `/admin` 解耦

### 后续可演进项

- WebSocket 实时日志推送
- 任务历史记录落盘
- 支持填写指定文档 URL 后触发拉取
- 支持选择“只部署 site 构建产物”还是“按 pipeline deploy fast”
- 增加终端 CLI 入口

## 验收标准

满足以下条件即可认为第一版完成：

- 本机打开 `/admin` 可看到控制台
- 页面可执行 `site/docs` 流程和 `pipeline` 流程的核心按钮
- 一次只能执行一个任务
- 日志和状态可见
- 部署按钮可把本地构建产物部署到 `121.199.175.111:/www/wwwroot/help.beta.ztocc.com`
- 有最小必要测试，并且在完成前已实际跑过
