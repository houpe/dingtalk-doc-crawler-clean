# help.beta.ztocc.com 钉钉鉴权部署说明

> 本文档记录文档站 `https://help.beta.ztocc.com` 的钉钉 OAuth 鉴权实现与线上部署信息。
> 部署或改动鉴权逻辑前，请先通读本文档，尤其是「⚠️ 重要约束」和「踩过的坑」两节。

## 一、架构概览

### 线上架构（2026-07-15 起）

```
浏览器
  │  https://help.beta.ztocc.com/...
  ▼
nginx (443, TLS 终止)
  │  proxy_pass → 127.0.0.1:4000
  ▼
Node 服务（pm2: docs-auth）
  ├── 钉钉鉴权层（未登录 → 302 /auth/login）
  ├── 登录页 /auth/login
  ├── 钉钉 OAuth 回调 /api/auth/dingtalk/*
  └── 通过后：express.static 服务静态文档
        （DOCS_DIST_DIR=/www/wwwroot/help.beta.ztocc.com）
```

**关键点**：鉴权发生在 Node 服务端。nginx 全量 `proxy_pass` 到 Node，由 Node 决定放行（已登录）或拦截（跳登录页）。VitePress 产物是纯静态文件，不参与鉴权。

**与纯静态部署的关系**：`/www/wwwroot/help.beta.ztocc.com` 仍是静态文档的源头。Node 服务通过 `DOCS_DIST_DIR` 环境变量直接读这个目录。更新文档内容时，仍用原来的 tar+scp 方式更新该目录，Node 服务无需重启即可读到新文件。

### 鉴权流程

**PC 扫码**：
```
访问页面 → Node 检查 session → 无 → 302 /auth/login
  → 用户点「使用钉钉登录」
  → GET /api/auth/dingtalk/url（生成 state，返回钉钉扫码 URL）
  → 跳钉钉扫码 → 回调 /api/auth/dingtalk/callback?code=&state=
  → 校验 state → getUserByCode → 写 session.user → save → 302 回原页
```

**钉钉 App 内**：
```
登录页检测到 dingtalk UA → 自动加载 jsapi → requestAuthCode
  → POST /api/auth/dingtalk/auto {authCode}
  → getUserByAuthCode → 写 session → 返回 success → 前端跳转
```

### 钉钉接口（lib/dingtalk.js）

- `getAccessToken()` — 企业 access_token，带缓存（提前 300s 过期）
- `getQrConnectUrl(redirectUri, state)` — 拼钉钉扫码 URL（OAuth2 v1.0）
- `getUserByCode(code)` — 扫码 code 换用户（含 unionId 富化真实姓名）
- `getUserByAuthCode(authCode)` — App 内 authCode 换用户（topapi 路径）

## 六、⚠️ 重要约束

1. **connect-redis 必须用 @7**，不能用 @9。@9 的 SET 对象语法 `{ expiration: { type, value } }` 不兼容 ioredis，会导致 session 写入报 `ERR syntax error`，表现为登录后死循环跳转。

2. **SESSION_SECRET 不要随意改**。已登录用户的 session cookie 是用它签名的，改了会导致所有现有 session 失效（用户需重新登录）。当前复用了 guide-backend 的 secret。

3. **nginx 必须传 `X-Forwarded-Proto $scheme`**。否则 Node 不知道请求是 https，`PUBLIC_ORIGIN` 仍能用，但 trust proxy 链路不完整。

4. **钉钉回调白名单必须包含** `https://help.beta.ztocc.com/api/auth/dingtalk/callback`。否则扫码后钉钉拒绝回调，登录失败。

5. **`/www/wwwroot/help.beta.ztocc.com` 不能删**。Node 通过 `DOCS_DIST_DIR` 直接读它。删了 = 站点空白。

6. **不要把 dist 重复上传到 /root/docs-site/docs/**。Node 不读那个路径，只用 `/www/wwwroot` 的。重复上传浪费空间且无意义。

## 七、踩过的坑（排查记录）

### 坑 1：connect-redis@9 + ioredis 不兼容

**现象**：扫码登录成功（日志有 `[auth] QR login: 姓名`），但 session 存不进 Redis，日志报 `[auth] session save error: ERR syntax error`，浏览器无限跳转循环。

**根因**：connect-redis@9 的 `set()` 用了 `client.set(key, val, { expiration: { type:"EX", value:ttl } })`，这个对象语法是 node-redis 客户端的，ioredis 不认识，会把它当原始参数发给 Redis，导致 `SET key val [object Object]` → `ERR syntax error`。

**修复**：降级到 `connect-redis@7.1.1`。@7 用 `set(key, val, 'EX', ttl)` 字符串语法，兼容 ioredis。

**代码兼容处理**：`auth-session.js` 用 `createRequire` 动态 require，取 `module.default || module.RedisStore`，兼容 @7（默认导出）和 @9（命名导出）两种导出形式。

### 坑 2：包未声明在 package.json

工作树里 `auth-session.js` import 了 `express-session`/`connect-redis`/`ioredis`，但 HEAD 的 `package.json` 没声明这些 dependencies（当时只在 devDependencies 里有部分）。全新 `npm install` 会缺包。修复时把这些加进了 `dependencies`。

### 坑 3：ESM 静态导入的版本差异

connect-redis@7 是 CJS 默认导出，@9 是 ESM 命名导出。直接 `import RedisStore from` 或 `import { RedisStore }` 都只能兼容一个版本。最终用 `createRequire` 动态 require 绕过，兼容两者。

### 坑 4：server.js 不加载 .env

`server.js` 直接读 `process.env`，不加载 `.env` 文件。pm2 启动时必须用 `--node-args="--env-file=/root/docs-site/.env"` 让 Node 20 内置的 `--env-file` 加载它。
